# Weekly Pattern Review

The point of a weekly review is not to grade the week. It is to spot patterns and adjust *one or two* things for next week. More than that and nothing changes.

## Output shape

```
WEEK OF <date>

Wins (specific, not generic)
- ...

Dropped balls (without shame)
- <thing>: what got in the way (in one phrase, not a confession)

Recurring obstacles
- The same thing that derailed me 2+ times this week.

Best focus windows
- Days/times when work actually happened.

Avoidance themes
- Categories I kept around but didn't touch.

Task-size mismatch
- Tasks that were actually projects in disguise.

Schedule realism score (1–5)
- Did the calendar match what was actually possible?

Routines
- Keep: ...
- Change: ...
- Remove: ...

Top 3 outcomes for next week
- 1.
- 2.
- 3.

Experiments for next week (small, falsifiable)
- "Try X for 5 days; notice what happens."
```

## Stance

- Lead with wins. Not as cheerleading — as data. The week was not zero.
- Treat dropped balls as information, not character. "Forgot" and "avoided" are different. Name which.
- One or two changes for next week, not ten.
- If the same obstacle shows up three weeks in a row, name it explicitly and propose a structural fix (not "try harder").

## When the user has files

If past weekly reviews exist (in the working directory or a notes folder the user has pointed to), read the last 3–5 and look across them. Surface trends the user might not see in a single week — recurring obstacles, recurring wins, recurring schedule slips. Cite specific lines.

## Do not

- Do not produce 2,000-word reviews. A useful review fits on one screen.
- Do not list every task that did not get done. Pick the ones that mattered.
- Do not promise next week will be different. Propose what to *try*.
