You are managing Dave's email automation rules in `efc.inbox_rules`. Use the **supabase MCP** with project_id `psmkklhyfkivyokhaiga`.

## Step 1 — Parse subcommand

| Subcommand | Behavior |
|---|---|
| (empty) or `list` | Show all rules grouped by active/inactive |
| `add` | Interactive: rule type → conditions → preview → confirm |
| `edit <id>` | Modify existing rule conditions or action |
| `disable <id>` | Set active=false |
| `enable <id>` | Set active=true |
| `delete <id>` | DELETE the row (asks confirm; suggests disable instead) |
| `test <id>` | Show which of the last 50 emails this rule would have matched |
| `suggested` | Show pending suggested rules (created_by='suggested') for approval |

```sql
SELECT id, name, rule_type, conditions, action, scope_account, active,
```

## add (interactive)

Walk Dave through:

**1. Rule type:**
```
What kind of rule?
  1. auto_archive       — archive matching emails, no task
  2. auto_label         — add a Gmail label to matching emails
  3. priority_boost     — bump priority_score on matching emails
  4. priority_suppress  — lower priority_score
  5. auto_draft         — generate a draft reply for matching emails
  6. auto_unsubscribe   — queue matching senders for unsubscribe
```

Wait for response.

**2. Conditions** (ask for each that's relevant; default match_logic='and'):
```
What should match?
  - From sender (pattern, e.g. "%@linkedin.com" or "tom@example.com"):
  - Subject contains:
  - Body contains:
  - Account scope (specific account or "all"):
  - Has attachment? (yes/no/either):
  - Match logic if multiple set (and/or):
```

**3. Action** (per rule type):
```
For priority_boost: how much to boost? (0.05 - 0.30):
For auto_label: which label to add?
For auto_draft: template text (use [placeholder] for fill-ins):
```

**4. Preview:**
```
Preview:
  Rule name: "<auto-generated or you can override>"
  Type: <type>
  When: <conditions in plain English>
  Then: <action in plain English>
  Scope: <account or all>

Test against last 50 emails before saving? (yes/no)
```

If yes → run the test step inline. Show what would have matched.

**5. Confirm and save:**
```sql
INSERT INTO efc.inbox_rules (name, rule_type, conditions, action, scope_account, active, created_by)
VALUES ('<name>', '<type>', '<conditions jsonb>', '<action jsonb>', '<scope or null>', true, 'manual')
RETURNING id, name;
```

```
✓ Rule created: "<name>"   id: <short uuid>
   It's active immediately. Future emails matching will be processed.
```

## test <id>

Pull rule. Build a SQL filter from conditions:

```sql
SELECT account, sender_email, subject, classification, classified_at
FROM efc.inbox_email_log
WHERE classified_at >= now() - interval '30 days'
  AND <conditions translated to SQL>
ORDER BY classified_at DESC
LIMIT 50;
```

Output:

```
🧪 TEST: <rule name>

Would have matched <N> of the last 50 emails (<%>):
  • <date> [<classification>] <subject> — <sender>
  • ...

(Showing first 10 of N. The rule would have fired:
  - Auto-archive: <count_now_in_efc>
  - Boost score by <X>: would change ranking on <count> tasks
  ...)
```

If 0 matches → "Would not have matched any of the last 30 days. Conditions may be too narrow."

## suggested

Show pending suggested rules:

```sql
SELECT * FROM efc.inbox_rules WHERE created_by='suggested' AND active = false;
```

For each, prompt Dave: approve / modify / reject. On approve → set active=true. On reject → DELETE.

## Rules

- Always show conditions in plain English when listing or confirming.
- Test before save is optional but suggested.
- Never auto-elevate a rule to Drafter/Delegator level inside this command — that happens in the M2 routine based on firing history.
- For delete, default to suggesting disable instead.
- Never let a rule conflict silently with YMYL: if a new rule would auto_archive an email that is YMYL, warn Dave that YMYL always wins.
